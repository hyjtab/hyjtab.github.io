OS的内存管理

注明：本系列实验基于彭东老师的操作系统实战课程，分享内容仅供作者本人留档阅读。
tips:此篇章将包含CSAPP中有关内存的篇章，一篇更比两篇强。

在之前的文章中我们谈到，CPU具有三种运行模式，分别为实模式，保护模式和长模式。其中在介绍长模式时，涉及到了分页这样一个名词，本部分重点介绍的便是分页、分段以及Cache等内存的重要机制。在介绍这种机制时，往往引入方法有很多，有自顶向下的设计，有自下而上的推导，我将尽可能将这一块干货极多也极为重要的部分讲清楚。下面我们就开始吧。

内存分级

首先明确一个基本事实，当我们的存储空间越大时，我们进行寻址的时间也就越长，如果不加处理地直接将CPU与硬盘相连，硬盘将不停地忙碌，而CPU将会不断的空等速度极慢的硬盘。存储器的这个特性，以及局部性原理的发现，设计者们采用了存储器分级的方式。现在我们来看现代计算机的分级存储结构：寄存器>一级Cache>2级Cache>3级Cache>主存>外存。寄存器能够直接被CPU操作，是最高级的存储结构。除了AX,BX等常见的程序用寄存器，有许多对程序员透明（不可见）的寄存器，比如之前说到的保护模式下的段寄存器的隐藏部分，这些设计大大加快了程序的运行效率。

Cache

其下就是多级Cache了。Cache使用的是SRAM，采用晶体管制作，不易受干扰，不需要刷新逻辑。它的优点就是——快！写入和读出都很快。但缺点也有——造价高，能耗大，集成度低——因此，我们选择它作为高速缓冲区的材料。而DRAM则不同，DRAM需要刷新，而且比较容易受干扰，速度会慢于SRAM，但优点也很明显——他的能耗低，集成度高，通常容量较大，所以它被用作主存。那么我们回到Cache。Cache分级，前两级每个单独的核心各有一个，最后一个大Cache则是所有核心共享的。其中涉及到了致命问题——写回与同步问题。当数据改变时，我们如何保证两个隔离开的高级Cache中的数据保持一致呢？答案是使用MESI协议。MESI协议是一个硬件协议，所以我们不具体介绍，总之，它能够维护高级Cache中的数据同步。值得一提的是高级Cache，也并不是结构相同的，一级Cache呼应了哈佛结构，将指令和数据分开存放，让程序运行更加高效。

接下来讲述一下Cache的工作方式。高速缓存器具有三种工作模式——全相联，组相联和直接映射，这三种方式的区别手段就是分组手段的不同——什么是分组手段？这里就要说明Cache中组、行、块、的概念了。Cache这一个整体，被整体上分为了若干行数相同的组，每个行又有相同大小的块。与接下来要介绍的虚拟内存一样的是，Cache也有映射的概念。而不同的是，由于Cache自身容量较小，所以会有未能映射的情况，由于速度较快，级别较高，也会有无效块——已映射无数据的情况，这时，在每行数据之前，就会有一个专门的标志位，标识这种情况。

随后讲述Cache中的寻址方式。我们需要明确的是，进入Cache此步后，已经进入寻址阶段，所以其地址已经是经过MMU处理（如果有虚拟地址的话）的了。目前的大部分计算机也是采用这种方式进行寻址的。那么我们针对MMU处理后的一个地址，如何在Cache中进行命中呢？这就要说到我们的分组机制了，Cache采用的寻址方式，便是，先命中组，再命中行，再命中块这样一个线性（也可能并行）的过程。Cache中组的编号一般都是按照物理顺序进行编码的，那么在翻译出的地址中，我们使用哪部分进行命中就是一个比较有趣的课题了。如果我们选择按照等级高低分配地址的高低，即最高几位被分配为组号，中间几位分配为行号，最后几位被分配为块偏移，那么我们一连串的数据组号便都是相同的。

这种方式听上去貌似还可以，但细想，则并不是最优解，甚至设计非常差。因为局部性原理，我们希望的是在一个程序执行过程中我们尽量提高Cache的命中率，就必须保证连续的数据同时存在在一个缓存区。如果我们以最高位做索引，则内存间相距较近的块被分到同一个组。而同一个组并不是能够让访存速度加快的方案，反之，由于Cache的编址特性，相同组号放在相同的位置，我们想要加载一个就近的组时，就需要先将Cache中我们之前访问的组卸载，然后再加载，如果我们以列的方式访问数组，如果数组每行的数据大小正好等于Cache编址时的行的长度，则会发生剧烈的抖动现象——每次Cache都不命中，这样的Cache使用起来反倒起负面的作用。但如果放在组的中间，我们就能够将相近的数据放在不同的组，充分利用数据的局部性。那么行号自然就是靠前的部分了。行号作为每一行的标识，我们必须找地方存储，所以就有了另外的标记位放在一行数据之前。

所以在Cache中，我们的地址被分为如下几部分处理——【标记位|组号|块偏移】，之前说过的可能并行的情况，便是一边匹配标记位，一边匹配组号，同时搜索对应值。回到全相联，组相联和直接映射。直接映射一组就是一行。这样会导致组号特别多，特别分散，命中率会相应的下降。那么全相联呢？是所有行被分到一组当中的情况。我们不需要处理组号，直接处理行标记就行。但这样也会带来一个缺点——我们的Cache成为一个巨大臃肿的整块，不仅不利于进程切换，而且大大增加了行标记匹配的难度，但在整体规模较小时，则速度要远快于其它两种方案，因此适合作TLB，也就是快表。而组相联则是每块中有若干行，是三级定位体系的标准使用者，具体寻址方式不再多说。

那么接下来说一说不命中的补救手段。对于直接映射，则是去除该组，重新加载，对于组相联，则是在对应组中选择一行删去，并加载，对于全相联，则是直接映射和组相联的结合，在全局中选择一行删去，并重新加载。

然后是写回策略。什么是写回策略呢？我们上述的策略中，仅仅讲到了读取数据的情况，如果我们要改变缓存中的数据，怎么确保同时改变了他在对应物理地址的改变呢？有两种方法：一种是直写——只要上层改了，不管三七二十一我就把这个修改传到下层，立马所有对应点全改，这种做法很符合直觉，但会引起巨大的总线流量。还有一种是写回，则与直写完全相反，尽可能慢的更新修改，直到该块被替换。对于写不命中，也有两种解决方案，一种是从低级存储器加载该单位到Cache，随后修改此值，另一种则是不加载该单位，直接在低级存储器中修改。前者被称为写分配，后者被称为非写分配。

在最后，我们来看如何具体的在操作系统中启动Cache。

在 x86 CPU 上开启 Cache 非常简单，只需要将 CR0 寄存器中 CD、NW 位同时清 0 即可。CD=1 时表示 Cache 关闭，NW=1 时 CPU 不维护内存数据一致性。所以 CD=0、NW=0 的组合才是开启 Cache 的正确方法。

mov eax, cr0;开启 CACHE 
btr eax,29 ;CR0.NW=0
btr eax,30 ;CR0.CD=0
mov cr0, eax

好了，我们的Cache部分就到此结束了。

主存管理

主存部分的管理，则是一个新的环节，在上节我们说到了三种模式，除了实模式之外，剩下的两种都貌似开始将我们本来平坦的储存区切割开来，有的不能访问，有的能访问，看起来十分繁琐复杂。但是这种设计正是为了一个东西的诞生——虚拟内存。
