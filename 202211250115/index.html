单向GBN的实现

首先，在进行介绍实现之前，我们应该充分理解什么是GBN协议：GBN协议，即Go Back N Step，也就是回退N步协议。同样基于滑动窗口，发送窗口可以有多个，但接收窗口只能有1个，且只能按顺序接受。窗口滑动的规则是，必须接受到接收方的确认后，窗口才可向前滑动。当出现丢包或无响应时，无论当前发到哪个包，将所有未确认的包全部重新上传。这基本上就是GBN协议规定的所有内容。具体可参见下图：

除了这些，重点要讲的是，将sendto和recvfrom包装到函数中时，不得不将使用的socket放到全局变量的位置。如果将socket提前初始化，将AE_INET,SOCKDGRAM等参数提前放置，则会导致socket生成失败，原因是因为在全局变量初始化时，并未加载winsock，这些宏定义有效，但不被windows处理。因而会造成socket的初始化失败。

recvfrom的有限阻塞问题

一般情况下，recvfrom是阻塞的。但由于windows设计问题，如果在一个udp包已经发送的情况下，如果此时发送者下线或不再发送回应，总之就是抛弃该连接了的话。会强制令recvfrom结束阻塞，WSAGetlasterror会返回报错代码10054。而这种情况下，我们无法维持recvfrom的阻塞，就不能正常进行线程计时器的管理。所以我们必须解决该问题。解决问题的办法如下述代码：

#pragma comment(lib, "ws2_32.lib")
#define SIO_UDP_CONNRESET _WSAIOW(IOC_VENDOR, 12)
  BOOL bNewBehavior = FALSE;
  DWORD dwBytesReturned = 0;
  WSAIoctl(sockClnt, SIO_UDP_CONNRESET, &bNewBehavior, sizeof bNewBehavior,
           NULL, 0, &dwBytesReturned, NULL, NULL);

其中，sockClnt需要换成当前程序使用的udp套接字。

recvfrom的计时问题

recvfrom的阻塞可以通过自身机制进行设置，注意，此法仅限用于默认的阻塞recvfrom不能用于设置成非阻塞状态的socket。具体做法如下：

timeval tv;
  tv.tv_sec = 1000;
  tv.tv_usec = 0;
  long b = 0;
  b = setsockopt(sockClnt, SOL_SOCKET, SO_RCVTIMEO, (char*)&tv, sizeof(tv));

mutex的使用问题以及std造成的小问题合集

可以通过#include < mutex >来使用信号量。生成信号量应该使用 std::mutex m1; 这样的语句，每一个信号量对象，都有几个方法，其中最常用的是lock和unlock。注意使用时尽量避免死锁发生。

系统设计思路

客户端设计

  • 发送函数：首先在窗口允许的范围内不停发送，窗口满之后，进行等待，待接收到服务器端的回应之后，滑动窗口，直到发送完毕。

  • 重发函数：发送函数判断当前是否处于重发状态，若处于，则移动重发指针，计算偏移，开启重发机制。同时需要注意各种线程问题。

  • 接收函数：注意线程问题，每个接收函数均置于不同的线程当中。负责移动接收指针，重发指针，开启重发，管理重发次数等线程级别交互问题。

  • 文件读取函数：将文件内容读到c风格字符串中。

  • 主函数：负责初始化和调用文件读取函数及发送函数。

服务器端设计

  • 接收部分：进行普遍嗅探型接收（为了更加直观的展示客户端行为，如果严格设置，则不易看到重发）

  • 发送部分：体现限制接收窗口，必须严格按序进行确认，在此处确认的包才可被视为进入服务器的包。

  • 生成序号函数： 为发送部分服务，生成正确的ack包编号。

包设计

第一字节被设计为序号位，第二字节被设计为确认位，第三字节被设为分段位，第四字节被视为段末位。
